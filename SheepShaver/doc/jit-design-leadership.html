1. Strategic Direction

Phase Strategy: C → B with FPU

"Our plan is to indeed start with option C and then move to option B with an FPU."

The user chose to start with the optimized interpreter (Option C) as a quick baseline, then transition to direct-codegen JIT (Option B) including full FPU support — bypassing the legacy dyngen architecture entirely.

Why Not Dyngen

The existing kpx_cpu JIT uses a QEMU-era dyngen approach (compile C templates, extract machine code from .o files). The user chose direct codegen instead because:

  • dyngen is compiler-version-fragile and architecturally dead
  • AArch64 ELF relocations (ADRP/ADD) aren't supported in dyngen
  • BasiliskII's working ARM64 codegen macros could be directly reused

2. Harness & Validation Workflow

"Build a scaffolding and a project plan in the repository, then sketch out a harness for opcodes, implementation, and start a test build."

Design Inputs

  • Test-driven from day one — the harness was built before the first opcode handler
  • Interpreter as oracle — run the same PPC bytecode under both interpreter and JIT, diff register dumps
  • SS_TEST_HEX / SS_TEST_DUMP — environment-variable-driven test injection, bypassing all ROM/GUI init
  • Determinism checking — each vector runs twice, both outputs must match
  • Self-contained test mode — allocates own RAM at low addresses (REAL_ADDRESSING safe), creates its own CPU instance

Harness Architecture

ComponentPurpose
SS_TEST_HEXHex PPC instruction sequence injected at startup
SS_TEST_DUMPEmit REGDUMP: with GPR0-31, CR, LR, CTR, XER
SS_TEST_JITCompile and execute natively instead of interpreting
SS_TEST_INITSeed initial register state

3. Tradeoffs Imposed

Hard Constraints

"No messing about with the ROM unless it's the absolute last resort. Prioritize fixing JIT opcodes."
  • No ROM patches to work around JIT bugs
  • No stub regions or RAM pre-sets
  • Booting is a progress marker, not a goal — opcode correctness is the objective
  • Commit and push regularly — no accumulating uncommitted work
  • Interpreter always available as fallback for uncompiled blocks

Coverage vs Correctness

"Our goal for today is 100% opcode coverage in the JIT harness."

The user pushed for breadth (all opcodes handled) while maintaining correctness (harness score=100). When partial implementations were found, the directive was clear:

"Implement all partials."

NOP Stubs: Explicit Justification Required

"And explain why you have NOPs."

Every NOP stub must have a documented justification. No silent shortcuts. The resulting NOP audit categorized 167 stubs across 8 categories with risk assessments.

AltiVec: Photoshop Matters

"Mac OS doesn't use AltiVec, but Photoshop does. Implement those."

The user pushed past "Mac OS boot coverage" to app-level compatibility, driving implementation of ~100 AltiVec→NEON handlers for real-world workloads.
