1. Architectural Vision

After requesting a structural audit of SheepShaver's codebase — its kpx_cpu PPC core, the dyngen JIT pipeline, and how it shares device-layer code with BasiliskII via symlinks — the user identified the key architectural question: whether to extend the existing dyngen JIT to AArch64 or replace it entirely.

Recognising that dyngen's template-extraction approach (compile C functions, parse the .o file for relocations) was both fragile and architecturally dead — QEMU itself abandoned it for TCG years ago — the user chose direct codegen, reusing BasiliskII's existing ARM64 instruction-encoding macros as the emission layer.

The Strategy Call

"Our plan is to indeed start with option C and then move to option B with an FPU."

This locked the roadmap: start with the optimized interpreter as a quick baseline (which benchmarking then showed was already 167 MIPS — fast enough for interactive use), then build the direct-codegen JIT incrementally on top, culminating in full FPU support. The user knew that the kpx_cpu interpreter's Duff's-device dispatch + block caching already provided a strong foundation, and that the JIT's value would come from eliminating the per-instruction indirect call overhead in hot loops.

2. Harness & Validation Design

The user directed that the test harness be built before the first opcode handler, establishing a test-driven workflow from the start. The design inputs were specific:

  • Interpreter as oracle — run identical PPC bytecode under both interpreter (jit false) and JIT (jit true), compare full register dumps. Any divergence is a JIT bug, not a test problem.
  • Environment-variable injection — SS_TEST_HEX injects PPC instruction sequences at startup, bypassing all ROM/SDL/GTK initialisation. This was a deliberate architectural choice: the test path intercepts in main() before any heavyweight init, allocates its own RAM at low addresses for REAL_ADDRESSING compatibility, and creates a standalone CPU instance.
  • Determinism as a gate — each vector runs twice, both outputs must match. Non-deterministic results are infrastructure failures, not opcode bugs.
  • Fuzzing for edge cases — the user directed addition of boundary-value vectors: integer overflow/underflow, shift counts at 0/31/32, divide by zero, MIN_INT/-1, carry chain propagation, FP signed-zero bit patterns, and sign-extension boundaries.
"Our goal for today is 100% opcode coverage in the JIT harness."

3. Tradeoffs & Constraints

No Workarounds

"No messing about with the ROM unless it's the absolute last resort. Prioritize fixing JIT opcodes."

This constraint — established during earlier BasiliskII JIT work and carried forward — prohibited ROM patches, stub regions, and RAM pre-sets as ways to mask JIT bugs. If the emulator misbehaves, the JIT handler must be fixed. This discipline was critical: during BasiliskII debugging, weeks had been lost chasing boot progress via ROM patches rather than fixing the actual opcode implementations.

Booting Is Not the Goal

"Booting is a progress marker, not the objective."

The user reframed the optimisation target away from "how far does it boot" toward "how many opcodes produce bit-exact results." This prevented the JIT work from drifting into the same boot-chasing pattern that had derailed the BasiliskII effort.

NOP Stubs Must Be Justified

"Explain why you have NOPs."

When the coverage report showed NOP-stubbed opcodes, the user required explicit justification for each one — categorised by architectural invisibility (cache hints), single-threaded safety (barriers), default-mode equivalence (FPSCR), and rarity (string ops). No silent shortcuts.

Partials Must Be Completed

"Implement all partials."

When the coverage report revealed simplified carry-chain implementations (ignoring XER.CA), the user directed that every partial be upgraded to a full implementation — including proper carry-in/carry-out through emit_read_xer_ca/emit_write_xer_ca_from_carry helpers.

Real Workloads, Not Just Boot

"Mac OS doesn't use AltiVec, but Photoshop does. Implement those."

The user pushed past boot-time coverage to application-level compatibility, recognising that AltiVec's vector operations map directly to ARM64 NEON and that Photoshop's pixel blending, alpha compositing, and filter kernels are the real-world use case for the emulator.
